 SECTION 1: CRITICAL SECURITY ISSUES
============================================================

1.1 [CRITICAL] Admin News CRUD — No Authentication
    Files: app/api/admin/news/route.ts (GET, POST)
           app/api/admin/news/[id]/route.ts (PUT, DELETE)
    Issue: All news API endpoints under /api/admin/news have ZERO
           authentication checks. Anyone can create, read, update, or
           delete news articles without logging in.
    Impact: Unauthenticated data manipulation, content defacement.
    Fix: Add JWT verification (isAdmin / getAdminPayload) to all handlers.

1.2 [CRITICAL] Maintenance Settings — Broken Auth (Token Existence Only)
    File: app/api/admin/settings/maintenance/route.ts
    Issue: The verifyAdmin() function on line 8-19 only checks if the
           user_token cookie EXISTS, it does NOT verify the JWT signature
           or check the role. Comment on line 16 confirms: "For now, we'll
           just check if it exists". Anyone with ANY cookie value can toggle
           maintenance mode on/off.
    Impact: Any user can enable/disable maintenance mode, lock out all
           visitors, or bypass maintenance protection.
    Fix: Replace with proper JWT verification + admin role check.

1.3 [CRITICAL] Profile Password Update — No Password Policy
    File: app/api/auth/profile/route.ts (PUT handler, line 53)
    Issue: When users change their password via profile update, there is
           no call to validatePassword(). A user with a strong password
           can change it to "123456".
    Impact: Bypasses the entire password policy we just implemented.
    Fix: Add validatePassword() check when password is provided.

1.4 [CRITICAL] Admin User Update — No Password Policy
    File: app/api/admin/users/[id]/route.ts (PUT handler, line 26)
    Issue: When admins update a user's password, there is no password
           validation. The password is hashed and stored directly.
    Impact: Admin can unknowingly set weak passwords for users.
    Fix: Add validatePassword() check when password is provided.

 SECTION 2: HIGH SEVERITY ISSUES
============================================================

2.1 [HIGH] XSS via dangerouslySetInnerHTML — 13 Components
    Files using dangerouslySetInnerHTML with DB/API content:
      - app/about-us/page.tsx
      - app/news/[id]/page.tsx
      - app/knowledge-base/[id]/page.tsx
      - app/admin/dashboard/Dashboard.tsx
      - app/admin/web-management/StaticPagesEditorPage.tsx
      - app/components/FloatingAnnouncement.tsx
      - app/components/Footer.tsx
      - app/components/StaticPageView.tsx
      - app/components/NewsSection.tsx
      - app/components/AnnouncementSection.tsx
      - app/information-center/page.tsx
      - app/projects/components/ModalContext.tsx
    Issue: Database-sourced HTML content (rich text from WYSIWYG editors)
           is rendered directly without sanitization. If an admin account
           is compromised or the admin news route is exploited (see 1.1),
           an attacker can inject malicious scripts.
    Impact: Stored XSS — steal session tokens, redirect users, deface pages.
    Fix: Sanitize HTML with a library like DOMPurify before rendering.
         Example: dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(content) }}

2.2 [HIGH] Chat/Send — No Rate Limiting + No Input Validation
    File: app/api/chat/send/route.ts
    Issue: Any unauthenticated visitor can send unlimited messages to any
           session by knowing the session ID. No rate limiting, no max
           message length, no input sanitization.
    Impact: Database flooding, storage exhaustion, potential DoS.
    Fix: Add rate limiting per IP, message length limits, and input
         sanitization.

2.3 [HIGH] Analytics Track — Unauthenticated Write Endpoint
    File: app/api/analytics/track/route.ts
    Issue: Completely unauthenticated POST endpoint that writes to the
           database. No rate limiting, no origin validation, no length
           limits on `path` or `session` fields.
    Impact: An attacker can flood the PageView table with millions of
           records, causing database bloat and performance degradation.
    Fix: Add rate limiting per IP, max length on fields, and optionally
         validate that the path is a valid site URL.

2.4 [HIGH] Chat/Status — No Authentication
    File: app/api/chat/status/route.ts
    Issue: Provides chat session details to unauthenticated users if they
           know the session ID. Information disclosure of chat messages.
    Impact: Visitor privacy leak.

2.5 [HIGH] NPM Dependency Vulnerabilities
    `npm audit` reports 10 vulnerabilities (2 low, 1 moderate, 7 high):
      - axios (1.0.0 - 1.13.4): DoS via __proto__ key in mergeConfig
      - flatted (<=3.4.1): Unbounded recursion DoS + Prototype Pollution
      - tar (<=7.5.10): Arbitrary File Read/Write via hardlink traversal
      - minimatch (<=3.1.3): Multiple ReDoS patterns
      - quill (=2.0.3): XSS via HTML export feature
      - @prisma/config (>=6.13.0): effect dependency issue
    Fix: Run `npm audit fix` for non-breaking fixes. For breaking changes,
         evaluate `npm audit fix --force` in a test environment.

2.6 [HIGH] Error Message Information Leakage
    Files: app/api/admin/repository/upload/route.ts (line 85)
           app/api/admin/stakeholders/post-document-all/route.ts (line 114)
    Issue: Error responses include `e.message` which can reveal internal
           server paths, database details, or stack traces to the client.
    Fix: Return generic "Server error" messages. Log details server-side only.

 SECTION 3: MEDIUM SEVERITY ISSUES
============================================================

3.1 [MEDIUM] Promote Endpoint — Timing Attack on Token Comparison
    File: app/api/auth/promote/route.ts (line 8)
    Issue: Uses direct string comparison `token !== expected` which is
           vulnerable to timing attacks. An attacker can determine the
           token character by character by measuring response times.
    Fix: Use crypto.timingSafeEqual() for token comparison.

3.2 [MEDIUM] No CSRF Protection on State-Changing Endpoints
    Issue: All POST/PUT/DELETE endpoints rely solely on JWT cookies.
           While SameSite=Lax provides partial protection, it doesn't
           cover all scenarios (e.g., top-level navigations with POST).
    Impact: Cross-site request forgery in certain browser configurations.
    Fix: Consider adding a CSRF token or using SameSite=Strict for
         admin cookies.

3.3 [MEDIUM] Maintenance Bypass — All /api Routes Excluded
    File: lib/maintenance.ts (line 92)
    Issue: shouldBypassMaintenance() excludes ALL /api routes from
           maintenance mode. This means all public-facing API endpoints
           continue to function during maintenance, potentially exposing
           data or functionality that should be restricted.
    Fix: Only bypass specific admin API routes, not all /api routes.

3.4 [MEDIUM] JWT in Non-Secure Cookie (HTTP) in Production
    File: app/lib/auth.ts (stakeholderCookieOptions)
    Issue: The Secure flag on cookies depends on FORCE_SECURE_COOKIES env
           variable, not on NODE_ENV=production. If this env var is missing
           in production, cookies will be sent over HTTP — exposing JWTs
           to network eavesdropping.
    Fix: Default to Secure=true when NODE_ENV === 'production', regardless
         of FORCE_SECURE_COOKIES.

3.5 [MEDIUM] SQLite in Production
    File: prisma/schema.prisma (line 13)
    Issue: Comment says "Switch to mysql in production" but the datasource
           is configured as sqlite. SQLite has limited concurrency, no
           network access control, and the database file could be exposed
           if file serving is misconfigured. The .gitignore properly
           excludes the DB file, but this is still a concern.
    Impact: Performance bottleneck under concurrent load, potential data
           exposure.

3.6 [MEDIUM] Admin/Users GET — Leaks User List to Any Authenticated User
    File: app/api/admin/users/route.ts (GET)
    Issue: getAdminPayload() checks for admin role, which is correct.
           However, the response includes all users' emails — consider
           if this data needs to be returned to all admins.

 SECTION 4: LOW SEVERITY / PERFORMANCE ISSUES
============================================================

4.1 [LOW] No Pagination on Admin List Endpoints
    Multiple admin API routes (news, employees, announcements, etc.)
    return ALL records without pagination. As data grows, this will:
      - Slow down API responses
      - Increase memory usage
      - Degrade admin panel performance
    Fix: Add pagination with cursor or offset-based patterns.

4.2 [LOW] PrismaClient Binary Targets
    File: prisma/schema.prisma (line 9)
    Issue: binaryTargets includes only "native" and "darwin-arm64".
           If the production server runs a different architecture
           (e.g., linux-x64), Prisma will fall back to native which
           may cause issues.
    Fix: Add "linux-musl-openssl-3.0.x" or "debian-openssl-3.0.x"
         to binaryTargets for the production server.

4.3 [LOW] Turbopack Build Warning — Missing nodemailer
    File: app/lib/email.ts
    Issue: The dynamic import of nodemailer generates a build warning.
           While handled gracefully, it adds noise to build output.
    Fix: Install nodemailer as an optional dependency, or use a
         conditional import pattern.

4.4 [LOW] Inconsistent Auth Helper Functions
    Issue: Authentication logic is duplicated across ~30 route files.
           Each file defines its own isAdmin(), getAdminPayload(),
           getInternalPayload(), etc. This creates maintenance burden
           and inconsistency (see 1.2 where the maintenance route has
           a different, broken implementation).
    Fix: Centralize all auth helper functions in app/lib/auth.ts and
         import them consistently across all routes.

4.5 [LOW] Database Connection — No Connection Pooling Config
    File: lib/prisma.ts
    Issue: While the singleton pattern prevents multiple PrismaClient
           instances, there's no explicit connection pool configuration.
           Under high load, this could cause connection exhaustion.
    Fix: Configure connection pool limits in DATABASE_URL or
         PrismaClient constructor.

4.6 [LOW] No Request Size Limits on JSON Endpoints
    Issue: Most POST/PUT endpoints call request.json() without size
           limits. An attacker could send very large JSON payloads to
           consume server memory.
    Fix: Configure Next.js body size limits in next.config.ts:
         experimental: { serverActions: { bodySizeLimit: '1mb' } }
